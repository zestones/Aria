## 4. Token overflow (RESOLVED) — context window overflow

### Symptom
```
BetaManagedAgentsUnknownError(
  message='prompt is too long: 203118 tokens > 200000 maximum',
  retry_status=BetaManagedAgentsRetryStatusTerminal(type='terminal'),
)
```
Returned via the SSE stream (~40 s into the session) after a successful burst of MCP tool calls (visible in backend access log as `172.25.0.6 → POST /mcp/.../ 200 OK`).

### Diagnosis

Managed Agents accumulates the full conversation (system + user message + every tool call + every tool result) in a single growing prompt. We pulled the actual session events for `sesn_011CaMSnFWeLhBCYkiqSLz8j` to find the offender:

| Tool call               | Input                                                 | Result size                     |
|-------------------------|-------------------------------------------------------|---------------------------------|
| `get_work_order`        | wo_id=7                                               | 614 chars                       |
| `get_equipment_kb`      | cell_id=…                                             | 2 147 chars                     |
| `get_current_signals`   | cell_id=…                                             | 969 chars                       |
| `get_signal_anomalies`  | 3h window                                             | 79 829 chars (~20k tok)         |
| **`get_signal_trends`** | **`signal_def_ids=[1,2,3,4], 3h window, 1m buckets`** | **2 307 202 chars (~575k tok)** |
| `get_logbook_entries`   | …                                                     | 13 chars                        |

**One tool call alone produced ~575 000 tokens** — almost 3× the entire context window. A 3-hour × 4-signal × 1-minute query should return ~720 rows; 2.3 MB suggests either:
- the bucketing helper is silently returning raw samples instead of bucketed averages, or
- the FastMCP wire envelope is ballooning the payload.

### Anthropic's recommendations (from the docs)

The Managed Agents docs explicitly mention the harness has “**built-in prompt caching, compaction, and other performance optimizations**” ([overview](https://platform.claude.com/docs/en/managed-agents/overview)). However:

- There is **no public `compaction` page** (the URL 404s) and **no API knob** to tune compaction thresholds.
- The session usage we observed proves caching IS active: `cache_read_input_tokens=13709`, `cache_creation_ephemeral_5m=2790`. So the harness was caching system prompt + early tool results.
- The 200k limit is the model's hard ceiling — compaction kicks in *before* that to summarize older turns, but a single tool result of 575k tokens **cannot be compacted away** because it exceeds the window in one event. Compaction works on the *history*, not on individual oversized responses.
- Therefore the docs implicitly require: **each tool response must individually fit — ideally well under 25k tokens** so the harness has room to keep multiple in context before compacting.

### Mitigations to try (in order of likely impact)

### Fix applied (`backend/aria_mcp/tools/signals.py`)

**`get_signal_anomalies`** — was returning one `dict` per raw sample (38 152 rows → 2.3 MB / ~575k tokens). Now aggregates consecutive breach samples into *breach windows*:
```
{'signal_def_id': 4, 'display_name': 'Discharge Pressure', 'breach_start': '…', 'breach_end': '…',
 'duration_seconds': 10547, 'threshold_field': 'low_alert', 'peak_value': 0.0, 'sample_count': 480, …}
```
Result: 38k rows → **28 windows, ~9 774 chars / ~2 400 tokens** (×240 reduction).

**`get_signal_trends`** — added a hard 500-row cap; appends `{"_truncated": true, "hint": "use 5m/15m aggregation or shorter window"}` when hit.
Result for the same 3h×4-signal query: **501 items, ~68 000 chars / ~17 000 tokens**.

Both verified live in the container; combined token budget for all tool calls is now well under 50k tokens.

## 4-old. Previous blocker (RESOLVED) — Anthropic stream hang

### Symptom
After successful `sessions.events.stream` opens, **no SSE events** are received. After 75 s the underlying `httpx` read times out:

```
httpcore.ReadTimeout
  File "agents/investigator/managed/events.py", line 80, in drive_session_events
    async for raw_event in stream:
```

### Diagnostics performed
- Initial user message + tools + mcp_server config all submitted successfully (no `400`).
- `docker logs aria-cloudflared` shows tunnel up and configured but no per-request logs (QUIC mode minimal logging).
- `docker logs aria-backend | grep -E 'mcp|POST' | tail -30` during the 75 s window: **zero non-localhost requests**. Anthropic's hosted MCP fetcher never hits our origin.
- Direct external curl to the same URL works in <100 ms, returns valid `initialize`.
- One earlier run did emit a `requires_action` with `sevt_01G1V6SocSS1BtH5yh1LuNwk` (see fix #6 above), suggesting the agent *can* sometimes start producing events — but the most recent runs hang from start.

### Hypotheses for follow-up
1. **Cloudflare WAF / bot-fight mode** silently challenging Anthropic's MCP fetcher user-agent. Worth checking the Cloudflare dashboard's Security → Events log for `aria-backend.vgtray.fr` for blocked requests around the test timestamps.
2. **Anthropic-side hosted MCP discovery timing out on first connect** but not surfacing the error on the SSE stream — instead the stream just stalls.
3. **Path-secret URL** (placeholder `change-me-to-a-random-32-byte-hex-token`) — unlikely to be rejected, but Anthropic may special-case obviously placeholder tokens. Worth swapping for a real 64-hex token (see §6 must-do).
4. **Beta header drift** — `managed-agents-2026-04-01` may have changed semantics since first integration. Worth pinning `anthropic` SDK and re-checking release notes.
